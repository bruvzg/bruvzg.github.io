Complex text layouts progress report #3

RichTextEdit:

Paragraph

Alignment tags ([left], [center], [right], [fill]) are combined with base direction, structured text override and language paragraph options to the single [p] tag. Standalone alignment tags are still available for compatibility.

[p align=x dir=rtl]

align left, center, right, fill Text alignment.
dir or direction ltr, rtl or auto Base text writing direction.
lang or language ISO language code Text language
bidi_override default, url, file, email, list, none, custom Structured text BiDi override for the paragraph.

Tables

Added support for setting cell background and border colors. Including alternating odd/even row backgrounds. Table horizontal layout is mirrored for RTL paragraphs.

[cell bg=odd_color,even_color border=color]AAA[/cell]

Additionally tables can be inlined into the text flow.

Lists

Added support for numbered and unnumbered lists (list tags existed before but were non functional).

[ol type=1]Item 1
Item 2
Item 3[/ol]

Following list types are supported:

  • Unnumbered lists ( [ul] ).
  • Numbered lists ( [ol] ) using numbers [1] (supports localised numbering systems, e.g. east Arabic numerals, based on paragraph language)
  • Numbered lists ( [ol] ) using Roman numerals: [i] and [I]
  • Numbered lists ( [ol] ) using Latin letters: [a] and [A].

Images

Added support for image inline vertical alignment:

[image align=top]

Font and outline

Added tags to control font size and OpenType features and font outline:

[font_size=36]

[opentype_features=scap]

[outline_color=red][outline_size=2]

or combined to the single tag

[font name=res://font.tres size=36 features=xxx]

Control characters

Unicode control character can be inserted to the text by inputing them directly (e.g. copy-pasting or using Godot input control context menu), or by using special tags (e.g. [rlm], [lrm], [zwnj]).

Dropcaps

Additionally, dropped capitals support is added to the RichTextLabel, dropcaps support same formatting options as the main text, and margins for precise placement control.

Designing custom controls:

In addition to the Text Server interface and draw function in the Font class, TextLine and TextParagraph helper classes, which serve as the base for most Godot controls, are available for more convenient single and multiline text rendering and designing custom controls.

TextLine and TextParagraph classes support following features.

- Multiple fonts and font options, and multiple languages in the single line or paragraph. Spans of text with designated font and properties can be sequentially pushed to the text buffer.

 
add_string(text, font, font_size, opentype_features, language);
 
- TextBuffer can embed user defined custom objects (e.g. images and tables, like RichTextLabel do) into the text flow. In the text object are represented as length object replacement characters, and follow all BiDi reordering rules.

add_object(key, size, inline_align, length);

Final object position in the shaped text layout can be retrieved by using

get_object_rect(key) function.

Both TextLine and TextParagraph provide direct access to the text server buffer RIDs to mix it usage with direct text server API calls.

Text server buffers, TextLine and TextParagraph use lazy calculation model, no BiDi reordering, shaping, line/word breaking or and other text processing is done unless it's necessary to do it for the requested get_ function call or buffer rendering. While layout text is immutable and can't be changed without redoing BiDi reordering and reshaping, layout size and alignment as well as embedded object sizes can be changed dynamically and done much faster than new buffer creation.

Limited vertical text support:

While Godot controls are designed to work with horizontal text only, Text Server and TextLine/TextParagraph have limited vertical text support as well. Vertical layout can be enabled by setting text buffer or helper class 'orientation' property.

Input:

Godot BiDi caret movement follows logical order of characters (right arrow always acts as forward direction), on the edge of RTL and LTR text two carets are displayed, primary caret  is determined base on last input direction (and can be switched manually using Ctrl/Cmd + ~). By default, all input except Backspace, which always deletes single character, moves caret over the whole graphemes (e.g. base character and combining diacritics), this behaviour can be changed by setting 'mid_grapheme_caret'

ICU data, build options and export templates:

Full blown CTL support is slower and result in bigger binary size, since some game project do not need it, multiple compile options are available to control text server building.

Following build options are available:

builtin_graphite, builtin_harfbuzz and builtin_icu (default value: True) - If true, uses builtin version of the libraries instead of system copies.
 
There are two text server implementations available by default:
- ICU + HarfBuzz + SIL Graphite 2 based text server with will complex text layout support.
- Fallback text server without CTL support, uses same API and provides roughly the same text rendering capabilities as in Godot 3.2.
 
Text servers can be enabled/disabled by using following build flags:
module_text_server_adv_enabled, module_text_server_fb_enabled
If editor/export template is built with multiple text servers, default one can be selected in Project Setting or by using --text-driver Name command line argument.
 
In the runtime text server can be changed by using TextServerManager class, but doing so will render all existing fonts and text buffers to become invalid. It's user responsibility to free all such resource (including default theme fonts) before switching Text Server.
 
Additional text servers can be implemented as C++ modules or GDNatvie libraries, loaded and used in the same way default servers are.

Some language do not use spaces between words, and dictionary (or other type rule base word breaking data) is required for better line breaking, ICU provides such data but it's size can be substantial for smaller projects. To reduce export size, ICU data is not included by default, but in can be added at export and stored in the PCK archive. It's always embedded into the editor binaries. Also you can avoid necessity for ICU data at all and manually adding valid breaking points (inserting zero-width breaking spaces, which can be inserted from Godot input controls context menu).
